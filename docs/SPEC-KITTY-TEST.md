# Spec-Kitty Integration Test Plan

## Purpose

Test whether [spec-kitty](https://github.com/Priivacy-ai/spec-kitty) (spec-driven development CLI) can work alongside our existing harness (ESLint + Steiger + Prettier + validate-architecture.ts + Storybook + Husky hooks + CI pipeline) in an FSD (Feature-Sliced Design) project.

**This is a compatibility experiment, not a commitment.** If friction outweighs benefit — we revert and keep our manual `docs/TICKETS.md` workflow.

---

## Background: What We Have

Our project is a self-enforcing FSD repository with these enforcement layers:

| Layer            | Tool                     | Command                   |
| ---------------- | ------------------------ | ------------------------- |
| Code quality     | ESLint 9 flat config     | `npm run lint`            |
| FSD architecture | Steiger                  | `npm run lint:arch`       |
| Formatting       | Prettier + TW plugin     | `npm run format:check`    |
| Type safety      | TypeScript 5.9           | `npm run build`           |
| Storybook gate   | Storybook build          | `npm run build-storybook` |
| Arch validation  | validate-architecture.ts | `npm run validate:arch`   |
| Pre-commit       | Husky + lint-staged      | auto on `git commit`      |
| Pre-push         | Husky                    | auto on `git push`        |
| CI               | GitHub Actions           | auto on push/PR           |

Key documentation files:

- `AGENTS.md` — vendor-agnostic agent instructions + workflow
- `ARCHITECTURE.md` — FSD layers, dependency graph (Mermaid), import rules
- `CONVENTIONS.md` — machine-enforceable rules with tool tags
- `CLAUDE.md` — Claude Code specific instructions
- `docs/TICKETS.md` — 17 implementation tickets adapted for FSD
- `docs/DDD_CONTEXT.md` — domain model mapped to FSD slices

## Background: What Spec-Kitty Does

**CRITICAL: spec-kitty is an ORCHESTRATOR, not a content generator.** CLI commands create directory scaffolds, YAML metadata, and git commits. The AI agent (you) writes the actual content — specs, plans, WP prompts. Don't expect spec-kitty commands to produce filled-in documents.

Workflow:

```
specify → plan → tasks (WPs) → implement (in worktrees) → review → merge
```

For each feature, it creates `kitty-specs/<NNN>-<feature-name>/` with:

- `spec.md` — **scaffold only**, agent fills requirements/scenarios/acceptance criteria
- `plan.md` — **template with ACTION REQUIRED placeholders**, agent fills technical approach
- `tasks.md` — kanban checklist, agent fills task breakdown
- `tasks/WP01-*.md` — work packages with YAML frontmatter (lane, deps, subtasks)
- `wps.yaml` — structured dependency manifest (auto-generated by `finalize-tasks`)

Each WP executes in an isolated git worktree under `.worktrees/`.

**Note:** Directory names are auto-generated and may differ from what you pass. Example: passing `"001-shared-money-value-object"` may produce `kitty-specs/001-001-shared-money-value-object/`. Do NOT hardcode paths — use `ls kitty-specs/` to discover the actual directory.

Install: `pip install 'spec-kitty-cli>=3.0.0'`

After install, verify: `spec-kitty --version` (must be ≥ 3.0.0)

#### Test 1.3: Create tasks / work packages and finalize

**Step 1 — create tasks.md and WP files:**

```

```

This is a slash command for AI agents. If running via CLI, you may need to create `tasks.md` and `tasks/WP01-*.md` manually.

**Step 2 — ensure WP frontmatter is correct:**

WP files MUST have YAML frontmatter with these fields (finalize-tasks validates them):

```yaml
---
id: WP01
title: Money Value Object
lane: backlog
dependencies: []
subtasks:
  - Create src/shared/lib/money.ts with Money class
  - Create src/shared/lib/money.test.ts with Vitest tests
```

**Critical format rules** (finalize-tasks will reject otherwise):

- Requirement refs must use inline format: `**Requirements Refs**: REQ-001` (NOT `### Requirement Refs`)
- `authoritative_surface` field is REQUIRED in frontmatter
- `lane` must be one of: `backlog`, `in_progress`, `for_review`, `done`

**Step 3 — finalize tasks:**


**Expected result:**

- Frontmatter enriched with `requirement_refs`, `owned_files`, `execution_mode`
- Files auto-committed by spec-kitty

**Verify:**

- Do WP files have complete YAML frontmatter?
- Do subtask file paths point to `src/shared/lib/money.ts` (FSD path)?
- Does guidance mention `npm run lint` and `npm run build`?
- Does the WP mention updating `src/shared/lib/index.ts`?

## Known Problems & Expected Friction

### Problem 1: ARCHITECTURE.md conflicts in parallel worktrees (CRITICAL)

**What:** Multiple worktrees each add edges to the Mermaid graph in ARCHITECTURE.md. On merge → conflict.

**When it triggers:** Tier 2+ tickets running in parallel (e.g., T-004 Cart + T-005 ProductVariant + T-006 Coupon simultaneously).

**Mitigation:** Merge WPs sequentially, not in parallel. After each merge, the next WP sees the updated graph.

**If this is too painful:** Consider switching validate-architecture.ts to auto-generate the graph from imports (instead of comparing to a hand-maintained graph). The script becomes the source of truth, ARCHITECTURE.md becomes output, not input.

### Problem 2: node_modules missing in worktrees (MEDIUM)

**What:** spec-kitty creates worktrees but doesn't run `npm install`. All npm commands fail.

**When it triggers:** Every `spec-kitty implement` that creates a new worktree.

**Mitigation:** Add to charter: "After entering a worktree, run `npm install` before any other command." Or add a post-worktree-create hook if spec-kitty supports it.

### Problem 3: Husky hooks may not fire in worktrees (MEDIUM)

**What:** Git worktrees share `.git` but Husky discovers hooks relative to the working directory. The `.husky/` directory exists in the worktree (it's part of the repo), but the hook runner may not find it.

**When it triggers:** `git commit` or `git push` inside a worktree.

**Mitigation:** Test during Phase 2.3. If hooks don't fire, rely on manual lint/build commands (already in our AGENTS.md workflow).

### Problem 4: spec-kitty paths vs FSD paths (MEDIUM)

**What:** spec-kitty's default templates may suggest non-FSD paths (e.g., `src/shared/domain/` instead of `src/shared/lib/`).

**When it triggers:** During specify/plan/tasks generation.

**Mitigation:** Strong charter with explicit path rules. If still wrong after charter — strengthen the spec prompt with explicit file paths.

### Problem 5: Over-engineering WPs for simple tasks (LOW)

**What:** spec-kitty may generate 5 WPs for a task that needs 1. Overhead of worktree per WP.

**When it triggers:** Small tasks like T-001 (Money VO), T-003 (Fixtures).

**Mitigation:** During `/spec-kitty.tasks`, tell the agent: "This is a small task, 1-2 WPs maximum."

## Success Criteria

| Criterion                                                    | How to measure                                                                  |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------- |
| spec-kitty specs use correct FSD paths                       | Read generated spec.md, plan.md — no non-FSD paths                              |
| All harness checks pass in worktree                          | `npm run lint && lint:arch && validate:arch && build && build-storybook` exit 0 |
| Agent follows our conventions (not just spec-kitty defaults) | Named exports, story-first, correct segments                                    |
| Merge back to main is clean                                  | No broken checks after merge                                                    |
| validate:arch catches missing graph edges                    | Intentionally skip ARCHITECTURE.md update → validator catches it                |
| Overhead is justified                                        | spec-kitty pipeline for 1 ticket takes ≤ 2x the time of manual implementation   |

## Decision Matrix

After running Phases 1-3:

| Result                                           | Decision                                                            |
| ------------------------------------------------ | ------------------------------------------------------------------- |
| All tests pass, overhead acceptable              | Adopt spec-kitty for remaining tickets                              |
| FSD paths wrong despite charter                  | Strengthen charter, retry once. If still wrong → reject             |
| Worktree friction too high (node_modules, hooks) | Reject worktree isolation, use spec-kitty specs only (no worktrees) |
| validate:arch conflicts unmanageable             | Reject parallel execution, use spec-kitty with sequential WPs only  |
| Overhead > 3x manual implementation              | Reject entirely, keep docs/TICKETS.md                               |
